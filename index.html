Tomcat：部署、配置与排错指南

1. 部署 Tomcat

访问 http://你的服务器IP:8080，看到 Tomcat 首页即成功。

💡 生产环境建议配置 systemd 服务：创建 /etc/systemd/system/tomcat.service

2. 主配置速览

  • 主配置文件：$CATALINA_HOME/conf/server.xml
  • Web 应用部署目录：$CATALINA_HOME/webapps/（放 WAR 包或目录）
  • 全局上下文配置：$CATALINA_HOME/conf/context.xml
  • 用户/角色配置：$CATALINA_HOME/conf/tomcat-users.xml

3. 报错怎么办？去哪找问题？

🔍 排错三步走

  1. 看 Tomcat 自身日志（最重要）

    日志目录：$CATALINA_HOME/logs/

日志文件 用途
catalina.out 主日志 + 应用 System.out 输出
localhost.log 应用启动/部署相关错误
manager.log 管理后台操作日志
catalina.<日期>.log 更详细的历史记录

  1. 检查应用部署日志

    • 部署 WAR 时，localhost.log 会记录解压、类加载失败等错误。
    • 访问应用 404，先看 webapps/ 下是否有对应的目录或 WAR 已展开。

  2. 确认端口是否监听

🚨 常见错误速查

现象 可能原因 去哪看
启动后访问 8080 无响应 端口被占用 / 防火墙未开放 catalina.out + netstat
404 应用未找到 WAR 未部署成功或路径错误 localhost.log + webapps/ 目录
403 Manager 页面拒绝访问 未配置用户/角色 tomcat-users.xml + manager.log
内存溢出 (OOM) JVM 堆太小 catalina.out 末尾，需调整 JAVA_OPTS
启动立即退出 JDK 未安装 / 环境变量错误 catalina.out 开头部分

4. 小贴士

  • 每次修改 server.xml 或 context.xml 后记得重启：shutdown.sh + startup.sh
  • 生产环境去掉 catalina.out 的冗长输出，改用 logrotate 切割。
  • 部署新 WAR 前，建议先删除 webapps/ 下对应的旧目录，避免缓存冲突。
  • 若需远程管理，务必修改默认密码并限制访问 IP。

Tomcat 问题都能从 catalina.out 和 localhost.log 里找到答案。

Nginx：部署、配置与排错指南

1. 部署 Nginx

访问 http://你的服务器IP，看到 Welcome 页面即成功。

💡 CentOS / RHEL 用 yum install nginx；Docker 方式：docker run -d -p 80:80 nginx

2. 主配置

  • 主配置文件：/etc/nginx/nginx.conf
  • 站点配置目录：/etc/nginx/conf.d/ 或 /etc/nginx/sites-enabled/
  • 日志目录：/var/log/nginx/（access.log + error.log）

3. 报错怎么办？去哪找问题？

排错

  1. 看 Nginx 自身报错

    • 启动/重载失败时，运行 sudo nginx -t 会指出第几行配置有误。
    • 服务起不来： systemctl status nginx` 看详细日志。

  2. 查错误日志（最常用）

    • 13: Permission denied → 文件权限/SElinux 问题
    • 111: Connection refused → 上游服务没启动
    • 24: Too many open files → 调整系统限制

  3. 确认访问日志（看请求是否到达）

常见错误

现象 可能原因 去哪看
502 Bad Gateway 后端服务挂了 error.log + 检查后端端口
404 Not Found root 路径不对 error.log 会记录文件路径
403 Forbidden 缺少 index 或权限 error.log + ls -l 检查目录
端口被占用 80/443 已被其他进程占用 sudo netstat -tulnp

ps

  • 每次改配置后务必 sudo nginx -t，避免重启失败导致服务断连。
  • 生产环境开启 error_log 级别 warn 或 error，避免日志撑爆磁盘。
  • 遇到诡异问题，先 sudo nginx -T 查看实际生效的全量配置。

学会看 error.log，你就解决了 90% 的 Nginx 问题。

4. 报错怎么办？去哪找问题？

🔍 排错三步走

  1. 查看服务端 NFS 状态

  2. 查系统日志（NFS 日志主要在 kernel 和 rpc 相关）

  3. 客户端测试连通性

🚨 常见错误速查

现象 可能原因 排查命令 / 日志位置
mount.nfs: Connection refused NFS 服务未启动 / 防火墙 systemctl status nfs-server；firewall-cmd --list-all
mount.nfs: Permission denied /etc/exports 未允许该客户端 exportfs -v；检查 IP 网段写法
mount.nfs: No route to host 防火墙阻止（portmap/nfs 端口） 开放服务：firewall-cmd --add-service=nfs --add-service=rpc-bind --add-service=mountd
客户端挂载后写文件报 Permission denied 共享参数 root_squash 或目录权限 cat /var/log/messages；服务端检查目录属主 ls -ld /data/share
Stale file handle 服务端目录被删除/重建 客户端 umount 再重新挂载

5. 小贴士

  • 修改 /etc/exports 后记得 exportfs -r，不用重启服务。
  • 生产环境建议用 sync 参数避免数据丢失。
  • NFS 依赖 RPC 服务（portmap/rpcbind），确保它已启动。
  • 防火墙开启 NFS 需要固定端口（附加配置 rpcinfo 相关），简单测试可先 systemctl stop firewalld。

✅ 90% 的 NFS 问题都在 /var/log/messages 和 exportfs -v 里找到线索。
